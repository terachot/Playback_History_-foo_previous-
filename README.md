# Playback History (foobar2000 component)

A lightweight foobar2000 component that adds a **Playback History panel** ‚Äî since foobar2000 has no built-in way to jump back to a previously played track, this fills that gap.

## Features

- **Tracks your last 20 played songs** automatically in the background.
- **Floating panel** (`Playback History`) showing the list, most recent at the top.
- **Double-click any track** in the list to jump straight to it.
- **‚óÄ Back button** ‚Äî steps back to the previously played track, one at a time, removing the current track from history as you go.
- Smart playback: if the track is still in your current active playlist, it plays from its existing position instead of adding a duplicate entry at the bottom.
- Built as a real **Default UI (DUI) Element**, so it inherits foobar2000's font settings automatically (`Preferences ‚Üí Display ‚Üí Fonts ‚Üí Lists`).
- History is session-only by design (cleared when foobar2000 closes).

## How to open it

The panel's menu command is auto-generated by foobar2000 itself (via the `ui_element_v2` popup-command flag) ‚Äî look for **"Playback History"** in foobar2000's menus. Since it's a proper DUI Element, you can alternatively dock it into your layout via **Layout ‚Üí Add New UI Element ‚Üí (Utility category) ‚Üí Playback History** instead of using it as a floating popup.

## Requirements

- Windows
- foobar2000 (Default UI)
- To build from source: Visual Studio 2022 (MSVC toolset v143), CMake ‚â• 3.15, and the foobar2000 SDK (`pfc`, `foobar2000/SDK`, `foobar2000/shared`, `foobar2000/helpers`, `foobar2000/foobar2000_component_client`) placed under `sdk/`

`CMakeLists.txt` locates the SDK relative to the repo root automatically (`${CMAKE_CURRENT_SOURCE_DIR}/sdk/foobar2000/SDK`) ‚Äî no machine-specific paths to edit. If your SDK genuinely lives somewhere else, override it at configure time instead of editing the file: `cmake -B build -DFB2K_SDK_PATH="C:/path/to/SDK"`.

## Manual patches applied to the SDK

The `sdk/` folder in this repo is a normal foobar2000 SDK checkout, but with two hand-edits already applied to make it build cleanly as a CMake + MSVC 2022 component. These are already committed here ‚Äî this section exists so that if anyone re-downloads a fresh copy of the SDK (e.g. to update to a newer version), they know what to redo:

1. **`sdk/pfc/pfc.vcxproj`** ‚Äî in the `<ClCompile Include="pfc-fb2k-hooks.cpp">` item, added `ExcludedFromBuild` entries for our actual build configurations:
   ```xml
   <ExcludedFromBuild Condition="'$(Configuration)|$(Platform)'=='Debug|x64'">true</ExcludedFromBuild>
   <ExcludedFromBuild Condition="'$(Configuration)|$(Platform)'=='Release|x64'">true</ExcludedFromBuild>
   ```
   Without this, `pfc.lib` and `foobar2000_SDK.lib` both define `pfc::crashHook()`/`pfc::winFormatSystemErrorMessageHook()`, causing `LNK2005: ... already defined`. The project already excludes this file for its own `"... FB2K"`-named configurations ‚Äî ours just aren't named that, so they needed adding explicitly. **If you also build for Win32/x86**, add matching `Debug|Win32` / `Release|Win32` conditions in the same block if you hit the same `LNK2005` there.

2. **`sdk/foobar2000/shared/shared.vcxproj`** ‚Äî bumped `<PlatformToolset>` from `v142` to `v143` (all configurations). Without this, MSBuild fails with `MSB8020: The build tools for Visual Studio 2019 (Platform Toolset = 'v142') cannot be found` on a VS2022-only machine.

No other `.vcxproj`/`.h`/`.cpp` files under `sdk/` have been modified ‚Äî everything else is stock SDK.

## Building

```powershell
# 64-bit (for 64-bit foobar2000)
cmake -B build -A x64
cmake --build build --config Release

# 32-bit (for 32-bit foobar2000)
cmake -B build-x86 -A Win32
cmake --build build-x86 --config Release
```

Output DLL: `foo_previous.dll` (in `build[-x86]/bin/Release/`). Copy it into your foobar2000 `components` folder (matching bitness) and restart foobar2000.

### Compile commands quick reference

CMake is the **only** build path this project actually uses ‚Äî there's no `build.bat`, `Makefile`, or manual `cl.exe`/`link.exe` step to maintain, everything below goes through `cmake`.

**Full build (first time or after changing `CMakeLists.txt`):**
```powershell
cmake -B build -A x64
cmake --build build --config Release
```

**Rebuild after only editing `.cpp`/`.h` files** (no need to reconfigure):
```powershell
cmake --build build --config Release
```

**Clean rebuild** (delete cached CMake state and start over ‚Äî do this if a fix to `CMakeLists.txt` doesn't seem to take effect):
```powershell
Remove-Item -Recurse -Force build
cmake -B build -A x64
cmake --build build --config Release
```

**Debug build** (same idea, different `--config`):
```powershell
cmake --build build --config Debug
```

**Build & copy straight to foobar2000** (one-liner):
```powershell
cmake --build build --config Release; Copy-Item "build\bin\Release\foo_previous.dll" "C:\Program Files\foobar2000\components\"
```

**If a linker error names a missing `.lib` that should exist** (e.g. `pfc.lib`, `shared.lib`, `foobar2000_SDK.lib`): these are built via `include_external_msproject` inside `CMakeLists.txt`, so a normal `cmake --build` should produce them automatically. If one is still missing (e.g. right after deleting its output folder by hand), build that one sub-project directly once to unblock:
```powershell
msbuild "sdk\pfc\pfc.vcxproj" /p:Configuration=Release /p:Platform=x64
```

**Common error ‚Üí fix:**
| Error | Likely fix |
|---|---|
| `Cannot open include file: 'pfc/...'` or `'SDK/...'` | Missing/wrong `target_include_directories` in `CMakeLists.txt` ‚Äî check `sdk/` and `sdk/foobar2000` are both listed |
| `cannot open input file 'X.lib'` | The sub-project that builds it isn't wired into `add_dependencies`, or the search path in `target_link_directories` doesn't match where it actually built (check the build log line showing the real output path) |
| A fix you already made keeps "reverting" | You likely edited the generated `build/*.vcxproj` by hand instead of `CMakeLists.txt` ‚Äî CMake overwrites that file on every reconfigure |

This made it straightforward to swap the panel's implementation (from a raw Win32 top-level window to a proper DUI popup element) without touching the history logic at all.

## Known limitations / open items

- **Cross-playlist playback**: if you press Back or click a track that isn't in the currently active playlist, it gets appended to whichever playlist is active right now rather than being played from its original playlist/position.
- **Button background theming**: the panel's background, text, and list colors follow foobar2000's active theme (including dark mode) live ‚Äî but the Back button's *background* stays native Windows style, since themed push buttons largely ignore custom background colors. Button text color still follows the theme.
- **No settings page**: font size and other preferences aren't user-configurable via `Preferences ‚Üí Tools` yet ‚Äî font size is inherited automatically from foobar2000's own list font setting, which covers the common case but isn't independently adjustable.
