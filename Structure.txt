📄 README.md
   ✓ What the component does (Playback History panel)
   ✓ Build instructions (x64 + x86)
   ✓ Known limitations / open items
   ✓ File-by-file structure notes

🔨 CMakeLists.txt
   ✓ CMake build system (Visual Studio 17 2022 generator, MSVC v143 toolset)
   ✓ Builds both x64 (`-A x64`) and Win32 (`-A Win32`) from separate build dirs
   ✓ Links pfc / shared / foobar2000_SDK via include_external_msproject
   ✓ Sets output filename to foo_previous.dll via OUTPUT_NAME

src/stdafx.h / stdafx.cpp
   ✓ Precompiled header pair
   ✓ Pulls in Windows headers + foobar2000 SDK headers
   ✓ Only extension.cpp / dllmain.cpp compile against the PCH (/Yu)
   ✓ history_manager.cpp / history_panel.cpp / settings.cpp / preferences_page.cpp
     also use /Yu (see CMakeLists.txt)

src/dllmain.cpp
   ✓ DLL entry point
   ✓ Minimal implementation, doesn't need editing

src/extension.cpp
   ✓ DECLARE_COMPONENT_VERSION only (name / version / description shown in
     foobar2000 → Preferences → Components)
   ✓ No menu command code here anymore — the "Playback History" menu entry
     is auto-generated by history_panel.cpp's ui_element_v2 popup flag

src/history_manager.h / history_manager.cpp
   ✓ Hooks play_callback_static to track the last 20 played tracks
   ✓ Owns all playback/navigation logic: play a track from history, "Back"
     (removes current track from history, plays the previous one)
   ✓ Checks the active playlist for an existing copy of a track before
     inserting a duplicate (falls back to appending if not found)
   ✓ No UI code — calls history_ui_element_instance::refresh_all() to push
     updates out to whatever panel instance(s) are currently open

src/history_panel.h / history_panel.cpp
   ✓ ui_element (entrypoint) + ui_element_instance (actual panel) pair
   ✓ Registered with KFlagHavePopupCommand → foobar2000 auto-generates the
     "Playback History" menu command (opens as a floating popup, or can be
     docked into a layout via Layout → Add New UI Element instead)
   ✓ ListBox (track list, double-click to play) + Back button
   ✓ Font size, background color, and button height read from settings.h —
     fall back to foobar2000's own theme/font automatically when set to
     "auto" (font size 0 / bg color -1)
   ✓ apply_settings_all() pushes new settings into every open panel instance
     immediately after Preferences → Apply, no restart needed
   ✓ Supports multiple simultaneously open instances (static registry)

src/settings.h / settings.cpp
   ✓ cfg_int declarations for the 3 user-adjustable values: font size (px,
     0 = auto), background color (COLORREF, -1 = auto/theme), Back button
     height (px)
   ✓ Persisted automatically by the SDK across foobar2000 restarts
   ✓ Central place to add more settings later without restructuring

src/preferences_page.h / preferences_page.cpp
   ✓ preferences_page_v3 (entrypoint) + preferences_page_instance (the page)
   ✓ Registered under preferences_page::guid_tools → shows up as
     Preferences → Tools → Playback History
   ✓ Font size + Back button height as numeric edit boxes; background color
     via checkbox ("use theme color") + Windows color picker
   ✓ Apply() writes to settings.h's cfg_int vars and calls
     history_ui_element_instance::apply_settings_all(); Reset() reverts the
     displayed (not yet applied) values to defaults per SDK convention


🚧 Known open items (see README.md for details)
   • Back/click-to-play can insert into the wrong playlist if the track
     isn't in the currently active one
   • Panel colors don't yet cover every control (Back button *background*
     specifically is limited by native Windows theming — button text color
     and everything else follows the theme/custom settings fine)

✅ Recently resolved
   • Font size, background color, and Back button height are now
     user-adjustable via Preferences → Tools → Playback History (see
     src/settings.* and src/preferences_page.*)
   • CMakeLists.txt no longer hardcodes a machine-specific SDK path — safe
     to commit to git and clone on another machine as-is (see README.md's
     "Manual patches applied to the SDK" section for the two sdk/ files
     that still need hand-editing if you ever re-fetch a fresh SDK copy)
